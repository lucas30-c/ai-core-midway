对，你抓到关键了：**每次请求都会走一遍“创建 ctx → 执行中间件链 → 进入 controller → 返回响应”**，所以“注入 ctx 到 controller”这件事也是**按请求发生的**。（`userService` 通常是单例注入，不一定每次新建；`ctx` 一定是请求级。）

你现在最难的是“数据流向/入口”，这特别正常——从 React 的“组件树数据流”切到后端的“请求链路数据流”，心智模型完全不同。

我用一个**你能类比 React 的模型**给你画清楚。

---

重点：ctx是由koa创建的
请求流程不是：

“controller 创建 ctx”
而是：

Koa 创建 ctx
→ Midway 把 ctx 注入 controller
→ controller 使用 ctx

ctx是根据请求而创建的
请求进来 → Koa 创建 ctx →
Midway 发现要调用某个 controller →
Midway 会：const instance = new APIController()
instance.ctx = ctx
（实际实现通过 IOC 容器 + 作用域管理）

也就是说：

ctx 是在“创建 controller 实例时”被注入的

@Inject() ctx: Context 这种注入代码，每个“需要用 ctx 的类”里写一遍就够了。
框架内部“注入机制” —— 只实现一次（框架做的）

Midway 的 IOC 容器内部实现了：

当发现某个类声明了 @Inject() ctx: Context
就在请求时把当前 ctx 注入进去。

这个机制你不用写，它是框架级能力。

# 1) 你这段代码里，哪些东西“每次请求都会变”？

```ts
@Inject()
ctx: Context;        // ✅ 每次请求都会是新的 ctx（请求级）

@Inject()
userService: UserService; // ❗通常是单例（全局共享），不是每次创建
```

Midway/Koa 的核心是：**请求级对象（ctx）+ 单例服务（service）**。

# 2) 项目的“入口”在哪里？

在 Midway 里你可以这样理解入口链：

### 入口 0：Node 进程启动

`npm run dev` 会启动 Midway，它会加载 `src/configuration.ts`。

### 入口 1：Midway 启动时做两件事

* **扫描你所有 `@Controller`、`@Middleware`、`@Catch`**
* 建立一套路由表 / 中间件链 / 全局异常处理

所以真正的“总入口”就是：

✅ `src/configuration.ts`

# 3) 一次请求的“数据流向”长什么样？

我们用你熟悉的“函数调用栈”方式画：

## 请求：`GET /api/get_user?uid=123`

### Step A：创建 ctx（请求级容器）

Koa 收到请求后，创建 `ctx`，里面装着 req/res/header 等。

### Step B：进入 Middleware 洋葱链

比如你挂了 `RequestIdMiddleware`：

* 先执行 middleware
* 给 `ctx.state.requestId` 赋值
* `await next()` 把控制权交给下一层

### Step C：路由匹配到 Controller 方法

Midway 查路由表：`/api/get_user` → `APIController.getUser`

然后做一件关键事：

> 创建一个 Controller 实例，并把 ctx 注入进去

也就是你的：

```ts
@Inject() ctx: Context;
```

此时 `this.ctx` 就指向“这次请求”的 ctx。

### Step D：调用 service（通常是单例）

`this.userService.getUser()` 执行业务逻辑。

### Step E：返回结果 → Midway/Koa 写回 res

你 `return { ... }`
框架会把它序列化成 JSON，写回响应。

### Step F：回到 middleware 的“end”

执行 `await next()` 之后的逻辑（比如日志、统计耗时）

---

# 4) 类比 React：它其实也像“树”，只是树的维度不同

React 里你熟悉的是：

> 父组件 props → 子组件 props → render

后端里更像：

> middleware（外层）→ middleware（内层）→ controller → service → return

把它类比成 React 的“高阶组件包裹”：

```txt
<RequestId>
  <Auth>
    <Controller />
  </Auth>
</RequestId>
```

* middleware 就像包裹 Controller 的 HOC/Provider
* ctx.state 就像 Provider 里挂的共享状态（但它是“每请求一份”）

---

# 5) 你现在可以用一句话记住数据流向

> **HTTP 请求进来 → 生成 ctx → 经过 middleware 链加工 ctx → controller 读取 ctx + 调 service → 返回结果 → middleware 收尾**

这就是后端版“数据流”。

# 6) 继续推进前的一个超实用练习（让你马上不迷糊）

你现在在 `RequestIdMiddleware` 里加一行：

```ts
ctx.logger.info({ step: "middleware", requestId: ctx.state.requestId });
```

在 controller 里也加一行：

```ts
this.ctx.logger.info({ step: "controller", requestId });
```

然后请求一次接口，你看日志顺序。

你会立刻“看见”数据流向。
